== **Advanced topics**

== **EXCEPTIONS**
=== What is an Exception?

At its core, an **Exception** is an event that occurs during the execution of a program that disrupts the normal flow of instructions.

Think of it as a runtime signal: when an error happens inside a method, the method creates an object—an **Exception Object**—and hands it off to the runtime system. This object contains information about the error, including its type and the state of the program when it occurred.

The primary goal of exception handling is to transition from a **fatal crash** to a **controlled detour**, allowing your application to log the issue, clean up resources, and continue running smoothly.

== Types of Exceptions in Java

Java categorizes exceptions into **three main types** based on how the Java compiler and runtime treat them. Understanding these types is crucial for writing clean, predictable code.



=== 1. Checked Exceptions (Compile-Time Exceptions)

**Checked exceptions** are conditions that a well-written application should anticipate and recover from. The Java compiler **strictly enforces** that these exceptions must be handled before the code can even compile.

* **Behavior:** If a method throws a checked exception, the calling code *must* either handle it using a `try-catch` block or declare it in its method signature using the `throws` keyword.
* **When they occur:** Usually during interactions with external systems, networks, or file systems where things outside of your code's control can go wrong.
* **Examples:**
** `IOException`: Failed network or file read/write operations.
** `FileNotFoundException`: Attempting to access a file on disk that does not exist.
** `SQLException`: Errors interacting with a relational database.

=== 2. Unchecked Exceptions (Runtime Exceptions)

**Unchecked exceptions** are conditions that usually occur due to programming errors, logic flaws, or improper use of an API. They extend the `RuntimeException` class.

* **Behavior:** The compiler **does not check** for these exceptions. You are not forced to wrap them in a `try-catch` or declare them with `throws`. They will compile perfectly fine, but they will crash the application at runtime if they occur.
* **When they occur:** Due to bugs in the code logic that should be fixed by the programmer rather than caught at runtime.
* **Examples:**
** `NullPointerException (NPE)`: Attempting to call a method or access a field on an object reference that is `null`.
** `ArrayIndexOutOfBoundsException`: Trying to access an array index that is negative or greater than or equal to the size of the array.
** `IllegalArgumentException`: Passing an invalid or inappropriate argument to a method.

=== 3. Errors

**Errors** represent severe, abnormal problems that occur at the system or JVM level.

* **Behavior:** Like unchecked exceptions, they are not checked by the compiler. However, **you should never attempt to catch or handle an Error**. They indicate fatal situations that the application cannot realistically recover from.
* **When they occur:** When the JVM completely runs out of resources or encounters a catastrophic failure.
* **Examples:**
** `OutOfMemoryError (OOM)`: The JVM has run out of heap memory and cannot allocate any more objects.
** `StackOverflowError`: A method calls itself recursively too many times, running out of stack space.

== The Java Exception Hierarchy

In Java, exceptions are organized in a strict object-oriented class hierarchy. Every exception, error, and runtime failure is represented by a concrete class. At the absolute top of this family tree sits the `Throwable` class.


=== The Structural Breakdown

==== 1. java.lang.Throwable
This is the **ultimate root class** for all exceptions and errors in the Java language. Only objects that inherit from this class (or its subclasses) can be thrown by the JVM or caught using the `try-catch` mechanism.

==== 2. java.lang.Error (Subclass of Throwable)
Errors represent catastrophic, fatal conditions that happen at the system or JVM level.
* **Rule:** Applications should **never** try to catch or handle an `Error`. If your application hits an Error, the best outcome is to let the program crash and log the system state.
* **Key Subclasses:** `OutOfMemoryError`, `StackOverflowError`, `VirtualMachineError`.

==== 3. java.lang.Exception (Subclass of Throwable)
This branch represents conditions that a reasonable, well-built application should anticipate and handle gracefully. The `Exception` class is split into two distinct paths:

* **Checked Exceptions:** Any class that directly extends `Exception` (and is *not* a subclass of `RuntimeException`) is a checked exception. The compiler forces you to handle these.
** *Key Subclasses:* `IOException`, `SQLException`, `ClassNotFoundException`.

* **Unchecked Exceptions (`RuntimeException`):** Any class that extends `java.lang.RuntimeException` forms the unchecked exception branch. These bypass compiler checks and represent logic mistakes.
** *Key Subclasses:* `NullPointerException`, `ArithmeticException`, `IndexOutOfBoundsException`.

=== Why the Hierarchy Matters for Code Design

Because exceptions are just standard Java classes, **polymorphism applies to them**.

When catching exceptions, you can catch a specific child exception or a broad parent exception. However, best practices dictate catching the most specific type possible:

[source,java]
----
import java.io.FileReader;
import java.io.FileNotFoundException;
import java.io.IOException;

class HierarchyPolymorphismDemo {
    public void demoCatchingOrder(String filePath) {
        try {
            FileReader reader = new FileReader(filePath);
        } catch (FileNotFoundException e) {
            // Good practice: Catches specifically what went wrong
            System.err.println("Specific handler: File missing!");
        } catch (IOException e) {
            // Catching the parent wrapper (Valid, but broader)
            System.err.println("General I/O handler: Error reading stream.");
        } catch (Exception e) {
            // Catching the ultimate parent (Catches everything, risky!)
            System.err.println("Catch-all handler for unintended runtime issues.");
        }
    }
}
----

== Handling Multiple Exceptions

In production applications, a single block of code inside a `try` block can often throw more than one type of exception. Java provides clean, efficient ways to handle multiple distinct errors without repeating code.

=== Approaches to Handling Multiple Exceptions

==== 1. Multiple `catch` Blocks
You can chain multiple `catch` blocks after a single `try` block. Each `catch` block handles a specific exception type.

* **Rule of Order:** You **must** catch the most specific (child) exceptions first, followed by the more general (parent) exceptions. If you catch a parent exception (like `IOException`) before a child exception (like `FileNotFoundException`), the compiler will throw an error because the child catch block is unreachable.

[source,java]
----
import java.io.FileReader;
import java.io.FileNotFoundException;
import java.io.IOException;

class MultiCatchDemo {
    public void executeFileOperations(String filePath) {
        try {
            FileReader reader = new FileReader(filePath);
            int data = reader.read(); // Might throw IOException
        } catch (FileNotFoundException e) {
            // Highly specific exception caught first
            System.err.println("Specific Error: The file could not be found.");
        } catch (IOException e) {
            // General parent exception caught second
            System.err.println("General Error: An I/O error occurred while reading.");
        } catch (Exception e) {
            // Ultimate fallback catch-all
            System.err.println("Unexpected Error: " + e.getMessage());
        }
    }
}
----

==== 2. The Multi-Catch Block (Java 7+)
If you want to handle different, completely unrelated exceptions using the **exact same logic**, you can combine them into a single `catch` block using the pipe (`\|`) operator. This significantly reduces code duplication.

* **Rule of Finality:** The exception variable in a multi-catch block (e.g., `e` in the example below) is implicitly `final`. You cannot reassign a new value to it inside the catch block.
* **Rule of Hierarchy:** You cannot combine exceptions in a multi-catch block if they share a direct parent-child inheritance relationship (e.g., `catch (FileNotFoundException | IOException e)` is a compile error because `IOException` already covers it).

[source,java]
----
import java.sql.SQLException;
import java.io.IOException;

class UnifiedMultiCatchDemo {
    public void processData() {
        try {
            // Code that might connect to a database or read a system configuration file
            System.out.println("Processing database and configuration actions...");
        } catch (IOException | SQLException e) {
            // Handles BOTH exceptions with the exact same recovery logic
            System.err.println("Critical Failure: System operation halted due to hardware or database failure.");
            System.err.println("Details: " + e.getMessage());
        }
    }
}
----

== The `finally` Block

The `finally` block is a key component of Java's exception handling mechanism. It contains code that is **guaranteed to execute**, regardless of whether an exception is thrown, caught, or completely absent.



=== Key Characteristics

* **Guaranteed Execution:** The `finally` block runs after the `try` block finishes normally, or after a `catch` block finishes processing an error.
* **Cleanup Duty:** Its primary real-world purpose is **resource management**—closing open files, network sockets, database connections, or releasing locks to prevent memory and system leaks.
* **Execution with Returns:** Even if a `try` or `catch` block contains a `return` statement, the `finally` block will still execute *before* the method actually returns the value to the caller.

=== The Only Exceptions to the Rule
The `finally` block will run almost 100% of the time, *except* in these extreme scenarios:
1. The JVM is forcibly terminated using `System.exit(0);`.
2. The underlying operating system crashes or loses power.
3. The thread executing the code is killed or interrupted unexpectedly.

=== Code Example: Traditional Resource Cleanup

[source,java]
----
import java.io.FileReader;
import java.io.IOException;

class FinallyDemo {
    public void readFileSecurely(String filePath) {
        FileReader reader = null;
        try {
            reader = new FileReader(filePath);
            int data = reader.read();
            System.out.println("Data read successfully.");
        } catch (IOException e) {
            System.err.println("Error encountered: " + e.getMessage());
        } finally {
            // This block ALWAYS executes
            System.out.println("Executing finally block for resource cleanup...");
            if (reader != null) {
                try {
                    reader.close(); // Crucial step to avoid resource leaks
                    System.out.println("FileReader closed successfully.");
                } catch (IOException e) {
                    System.err.println("Failed to close reader: " + e.getMessage());
                }
            }
        }
    }
}
----

=== Modern Alternative: Try-With-Resources (Java 7+)

While the `finally` block is great for general cleanup, using it manually to close resources can lead to bulky, nested `try-catch` structures (as seen above). Modern Java code prefers **Try-With-Resources**, which implicitly creates a hidden `finally` block to close anything implementing `AutoCloseable`.

[source,java]
----
import java.io.FileReader;
import java.io.IOException;

class TryWithResourcesDemo {
    public void streamlinedReadFile(String filePath) {
        // The resource is declared inside the try parentheses.
        // Java automatically closes it at the end, eliminating the need for a manual finally block.
        try (FileReader reader = new FileReader(filePath)) {
            int data = reader.read();
        } catch (IOException e) {
            System.err.println("Error: " + e.getMessage());
        }
    }
}
----

== Throwing and Re-throwing Exceptions

Exceptions don't just happen automatically; you can deliberately control the flow of your application by manually initiating exceptions or passing them up the call stack to a component better equipped to handle them.



=== 1. Throwing an Exception (`throw`)

The `throw` keyword is used to explicitly clear the normal execution path and hand an exception object over to the runtime system. This is typically used when validation checks fail or a method detects that its preconditions have been violated.

* **Behavior:** Once a `throw` statement is hit, execution of the current method stops immediately. The JVM looks down the execution stack for a matching `catch` block.

[source,java]
----
class Account {
    private double balance = 100.0;

    public void withdraw(double amount) {
        if (amount > balance) {
            // Manually creating and throwing a runtime exception
            throw new IllegalArgumentException("Insufficient funds! Cannot withdraw: " + amount);
        }
        balance -= amount;
    }
}
----

=== 2. Declaring Exceptions (`throws`)

If a method throws a *checked* exception, it must advertise this fact in its signature using the `throws` keyword. This forces any caller of this method to either wrap the call in a `try-catch` block or declare the exception themselves.

[source,java]
----
import java.io.IOException;

class DataExporter {
    // The throws keyword acts as a warning sign to any method calling this one
    public void exportData(String data) throws IOException {
        if (data == null) {
            throw new IOException("Export failed: Data payload cannot be null.");
        }
        System.out.println("Exporting data stream...");
    }
}
----

=== 3. Re-throwing an Exception

Sometimes, a method catches an exception but cannot completely resolve the issue itself. You can catch the exception, perform local actions (like logging or rolling back a local transaction), and then use the `throw` keyword again to pass that same exception—or a brand new one—up to the calling method.

* **Chained Exceptions:** When throwing a new exception from inside a catch block, it is best practice to pass the original exception into the constructor of the new one. This preserves the root cause stack trace.

[source,java]
----
import java.sql.SQLException;

class DatabaseManager {
    public void executeQuery() throws RuntimeException {
        try {
            // Simulating a database failure
            throw new SQLException("Connection timed out.");
        } catch (SQLException e) {
            // Local Action: Log the specific technical database error
            System.err.println("Database error logged locally: " + e.getMessage());

            // Re-throwing a clean, business-level abstraction wrapper
            // Passing 'e' ensures the original stack trace trace isn't lost
            throw new RuntimeException("Application failed to fetch user record.", e);
        }
    }
}
